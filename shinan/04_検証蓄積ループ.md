# 検証蓄積ループ

## 目的

サキヨミAIを、検証するほど濃くなる仕組みにする。
チャット上の感想で終わらせず、1レース単位で保存し、負け方と穴逃しを次の改善に変換する。

## 基本方針

```text
予想ログ
→ 結果ログ
→ 1レース台帳
→ 穴逃しログ
→ 診断レポート
→ 改善仮説
→ コード修正
→ 別月で再検証
```

この順番を守る。
勝った月だけで修正しない。
修正したら、必ず別月または別seedで検証する。

## 保存するもの

### 1. summary.json

各検証バッチの集計元。
買い目、結果、的中、払戻、見送り理由を含む。

### 2. 検証データ台帳.jsonl

1レース1行の研究用データ。
ここを増やすほど、会場別・星別・見送り理由別の分析ができる。

### 3. 穴逃しログ.md

1万円以上の払戻を逃したレースだけ集める。

分類:

- 見送り穴
- 購入外れ穴
- 星5見送り穴
- HUMAN_CHECK穴
- アレド厳格化で落とした穴

### 4. 検証診断レポート.md

50Rまたは100Rごとに生成する。

見る項目:

- 全体回収率
- 購入率
- 的中率
- 星別成績
- 判定別成績
- 会場別成績
- 見送り穴の理由
- 購入外れ穴の形

## 100Rごとの判断ルール

### 回収率100%以上

まだ勝ち確定ではない。
一撃依存か、複数的中で勝っているかを見る。

### 回収率80〜100%

惜しい。
点数削減、オッズ番人AI、展示反映で改善できる可能性が高い。

### 回収率50〜80%

ロジックの方向は残しつつ、買う条件を絞る。
穴逃しログを優先して見る。

### 回収率50%未満

買い目生成またはゲート設計に大きな穴がある。
同じルールで追加検証する前に、負け筋を分解する。

## 修正してよい条件

- 50R以上で同じ負け筋が出ている。
- 穴逃しログに同じ停止理由が複数出ている。
- 勝ち筋が1本の高配当だけに依存していない。
- 修正後、別月で再検証できる。

## 修正してはいけない条件

- 1レースだけの結果に合わせる。
- 勝った日の結果だけでルールを強める。
- 払戻や人気を予想入力に混ぜる。
- 過去オッズを締切前オッズとして扱う。
